Implementation

There are two key features paramount to the web application: the display websites in frontend, and the explainer module in backend. In this part we will go through how these two features are implemented and what tools and approaches we utilised to do so.

Automating Implementation

While implementing these key features, we also used sentry-sdk to catch exceptions occurring in production, allowing issues to be caught efficiently. Error information are also sent to sentry, adding even more relevant data, making incident reporting and response easier. Dealing with other forms of software bugs and errors are elaborated in the testing page of this report website.

 

Display Websites

The websites are implemented using Django, Bootstrap abd Azure Web Services. We chose DJango for its high scalability as well as its versatile nature, with high security to boot. 

We also used Django REST framework to create REST API endpoints for models with auto generated documentation, which was crucial for changing the parameters of machine learning application for users.

To facilitate better CI/CD practices, we also utilised GitHub Action’s CI feature for pipelining the application’s building and testing without human intervention. Github Action’s CI consistently delivers build results and enabled us to merge pull requests with reliable backups.

Azure Web Services & Azure Databases were used to host the web app's frontend webpages live. Several Azure commands and dependencies have been incorporated into the project for these features to be integrated into the web application.

While coding the display website, we adopted a classic Model-View-Controller(MVC) architecture as it supports rapid and parallel development.  If an MVC model is used to develop any particular web application, then it is possible that another developer can work on the view while another works on the controller to create the workable logic of the web application. In other words, modifications to views would break our API, and similarly otherwise. 

Django in general already follows the SOLID Principles, but here are some examples on how we utilised these features.

Single Responsibility Principle - SSR states that a class should only have a single responsibility amongst multiple modules of the application. For example, we explicitly declared 2 separate model classes, a “Dataset” class and a “Post” class, both are used in the same page, but changing elements from one class does not disrupt the other.

Liskov’s Substitution Principle - every subclass or derived class should be substitutable for their base or parent class. Django uses this through their template system. The master template named "base.html" defines an HTML skeleton document that is utilised amongst other templates. The "extend" tag tells the template engine that this template is inheriting from another template superclass. This allows us to share multiple properties among pages to increase page conformity and eliminate redundancy.

In addition to the above principles, we also utilised an Abstract Factory Design Pattern on class "Author" in the file models.py to ensure encapsulation and provides the user a way of creating objects of related classes(e.g "Dataset" & “Post” classes etc) at a given instance without exposing the exact class that is being instantiated at runtime.

Explainability Module

The explainability module is implemented through a combination of essential libraries. We used Joblib, Keras and Tensorflow to load the models, while Joblib, Pandas and Numpy are used to load the datasets. Finally, we used the open source Interpret Community SDK to generate the explanations and graphs for models and datasets.

Out of the available algorithms in Interpret Community SDK, we chose to use the build-in SHAP functionalities and its 4 different explainers using SHAP: the SHAP DeepExplainer for deep neural networks models, the SHAP TreeExplainer for tree based models, the SHAP LinearExplainer for linear models and the SHAP KernelExplainer for other models.
