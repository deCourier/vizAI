Testing

Testing is divided into two parts: compatibility testing and automated unit and integration testing. We conducted these tests in various ways from manually testing functions to using automated tools to ensure that every aspect of the prototype application is tested.

Compatibility Testing

Backend

Informal compatibility testing was done consistently on the backend code in order to understand the intricacies of how to use the Interpret Community SDK. This involved checking which types of models could be explained, which method of saving and loading was easiest to work with, etc. In the end we found that although many methods of storing the models work, pickling them with the joblib library was fastest and most consistent, and worked with the widest range of models.

During testing, we discovered that our self-trained models using Azure's proprietary Designer and Automated-ML tools are incompatible with Interpret-Community SDK. We decided that in order to gather future test data we should find industry sourced models, collecting the models from sites such as Kaggle. This way we gather data quicker, can test our data against real life models and can test a variety of different types of datasets from different industries in order to prove that it has a generalised usage. We used these models as the data for the integration tests.

Frontend

Since the frontend is an HTML website connected to the backend running on a cloud computing service, we performed manual tests to check if different webpages can load and function correctly across multiple platforms. Multiple types of devices running different operating systems have been used for integration and compatibility testing:

From this, we concluded that the frontend website is at least compatible with major operating systems running on popular desktop and mobile devices.

Automated Unit and Integration Testing

Automated unit testing on the prototype application was done using a test suite provided by Django. The unit testing framework was inspired by JUnit, and supports test automation, sharing of setup and shutdown code for tests, aggregation of tests into collections, and independence of the tests from the reporting framework.

Four types of automated tests are performed and passed successfully:

Test NamePurpose
tearDownTear down test on the whole system
test_filesTesting whether descriptive errors are raised successfully
test_context_managerTesting the integrity of context manager
test_file_modeTesting whether a bad mode can be set accidentally by the system
